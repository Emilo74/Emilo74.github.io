Intro to Software Analysis

Software analysis is the process of automatically discovering useful facts about programs. Software analysis can be broadly classified into three categories:

  • Dynamic (run-time)
  • Static (compile-time)
  • Hybrid (combination of dynamic and static analysis)
The effectivess of a software analysis tool can be measured by its soundness and completeness.

Dynamic analysis tools gather information during run-time. These analyses discover information by running the program and observing its behavior. Since a dynamic analysis gathers information by running the program, its cost is proportional to the execution time of the program. A longer run costs more than a shorter one. A dynamic analysis can achieve completeness - it is possible to design an analysis that will only report errors that can be triggered in a concerete execution of the program. It may miss some errors, as it inspects only a finite number of runs whereas the program may contain an unbounded number of paths. We say that a dynamic analysis is unsound; it may say that an incorrect program is correct.

Static analysis tools gather information during compile-time. These analyses discover information by analyzing the program's source or binary code. The cost of running a static analysis tool is proportional to the size of the program's source code. A larger program costs more than a smaller one. A static analysis can achieve soundness - it will not miss errors, but it may report spurious issues. Static analysis is incomplete; it may say that a correct program is incorrect.

Hybrid analysis combine aspects of both dynamic and static analyses by combining run-time and compile-time information in interesting ways.

A sound analysis means we can trust the correctness of its output.

If it says a program is correct, then the program is definitely correct. However, the analysis may improperly say that some correct programs are not correct. A trivially sound analysis returns false for all inputs. In short, a sound analysis never says an incorrect program is correct. It may, however, say that a correct program is incorrect (in other words, incomplete).

A complete analysis means we can trust that what it excludes from its analysis is incorrect.

If it says a program is not correct, then the program is definitely not correct. However, the analysis may improperly say that some incorrect programs are correct. A trivially complete analysis returns true for all inputs. In short, a complete analysis never says a correct program is incorrect. It may, however, say that an incorrect program is correct (in other words, unsound).

A software analysis tool cannot be both sound and complete - at least, not if we want the analysis to terminate. Designing a program analysis tool is an art that involves striking a suitable tradeoff between termination, soundness, and completeness.

Intro to Software Testing

Software testing checks whether a program's implementation agrees with its specification. Without a specification, there is nothing to test. A developer writes the implementation, while the tester writes the specification. In order for testing to be effective, both the specification and the implementation must be correct. Software testing is an art and a science. Development resources are finite, so there is a limit to how many tests can be written for a given piece of software. A specification can also evolve over time, so the test suite must be updated alongside it. Approaches to testing can be classified according to two orthogonal axes:

  • Manual vs Automated
  • Black-box vs White-box

Testing approaches are not stricly black-box or white-box. Likewise, testing need not be fully manual or fully automatic. It is better to think of these axes as continua instead of discrete categories. Testing that requires more human direction falls closer to the manual side of the axis, while testing that is performed primarily without human direction falls closer to the automated side of the axis. Black-box testing refers to testing where the tester can see nothing about the tested program's internal mechanisms and logic. The tester can only issue inputs to the program and observe the program's outputs. Program outputs are compared against the specifications required of the program. White-box testing refers to testing in which the internal details of the program being tested are fully available to the tester. In this case, the tester can use these internal details to perform a more precise analysis of the tested program and uncover inputs that are likely to trigger errors.
